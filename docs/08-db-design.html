1 設計方針

設計前提 必須

本課題のサイト本体は HTML / CSS / JavaScript による静的サイトとして実装しており、現在は JSON と localStorage を用いてデータを管理している。

ただし、将来的に本物の Web アプリケーションとして運用することを想定し、会員情報、商品情報、ハムスター情報、注文情報、景品情報などをデータベースで管理できるように設計を行う。

本設計書では「はむのくらし」をハムスターカフェ、ハムスター生体販売、関連商品の EC、会員制度を持つサービスとして想定し、必要なテーブルと関連を整理する。

管理対象データ 必須

管理対象は、会員、商品、商品カテゴリ、ハムスター、注文、注文詳細、カート、カート内商品、景品、ポイント履歴の 9 系統とする。

これにより、商品購入、会員管理、景品交換、ハムスター一覧表示、購入履歴の確認まで一貫して扱える構成とする。

    └ rewards

 └ products

hamsters

rewards
ER図の説明 必須

会員はカート、注文、ポイント履歴と関連し、EC と会員制度の中心となる。

商品は商品カテゴリに属し、カート内商品と注文詳細から参照される。

ハムスター情報は商品とは分離し、生体販売用の一覧・詳細ページで使用する独立エンティティとして扱う。

景品はポイント履歴と関連し、会員がどの景品へ交換したかを管理できるようにする。

5 初期データ

初期データ方針 必須

商品カテゴリには「ケージ」「餌」「おもちゃ」「ケア用品」「スターターセット」を登録する。

商品テーブルには現在の JSON データに合わせて約 26 商品を登録し、画像パスはローカルファイル名に対応する値を保持する。

ハムスターテーブルには 7 種類を登録し、種類名、性格、飼いやすさ、価格、説明、画像パスを保持する。

景品テーブルには「小さな遊び道具」「アクリルキーホルダー」「限定トートバッグ」「限定ぬいぐるみ」を登録する。

備考 必須

本課題では実際のデータベースは実装していないが、本設計をもとに Spring Boot と MySQL などを導入すれば、会員認証、注文管理、商品管理、景品交換履歴管理へ発展できる構成となっている。
